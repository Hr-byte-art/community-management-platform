## 3. 本次新增的扩展内容

### 3.1 数据库总脚本整合

新增并完善了数据库总脚本：

- `houduan/src/main/resources/sql/community_service.sql`

该脚本已经整合以下内容：

- 当前项目所需全部表结构
- 演示数据
- 索引
- RBAC 权限初始化数据
- 工单超时演示数据
- 公告订阅相关表与示例数据

扩展价值：

- 后续初始化数据库时，不再需要按 Sprint 分批执行多个 SQL
- 支持直接重建完整项目数据库
- 降低环境搭建和回归验证成本

### 3.2 工单超时治理前端收口

工单管理页面补充了超时治理所需的关键入口：

- 新增超时状态筛选
- 新增截止日期区间筛选
- 新增责任人筛选
- 列表增加责任人、截止时间、超时状态展示
- 新增/编辑/处理弹窗补充截止时间与责任人字段

扩展后，工单超时治理不再只是“首页看板能看见”，而是已经真正进入主业务页面，可支持：

- 管理员按截止日期查找工单
- 查看哪些工单已超时
- 对工单设置责任人和处理时限

### 3.3 自动派单规则预览

派单规则页接入了“规则预览”功能：

- 前端页面新增预览弹窗
- 接入后端 `/dispatch/rule/preview` 接口

扩展后，管理员在维护规则时，可以先输入工单类型和优先级，预览最终会命中哪个责任人，降低配置错误率。

### 3.4 公告统计与公告订阅设置

公告页扩展了两项能力：

1. 公告统计
- 前端新增“统计”入口
- 对接后端 `/notice/{id}/stats`
- 可查看浏览次数与阅读人数

2. 公告订阅设置
- 前端新增订阅设置弹窗
- 支持用户按 `NOTICE / ANNOUNCEMENT / NEWS` 三类公告类型配置订阅偏好

### 3.5 公告发布后的订阅推送闭环

本轮补齐了公告订阅的后端闭环逻辑：

- 新增 `NoticeSubscriptionController`
- 完成订阅配置读取接口
- 完成订阅配置保存接口
- 公告发布后，按订阅设置向用户发送站内消息

扩展后形成的业务链路为：

用户配置公告订阅 -> 管理员发布公告 -> 系统筛选订阅用户 -> 用户在消息中心收到“您订阅的通知已发布”消息

### 3.6 用户管理前后端一致性修复

本轮修复了用户管理中的一个关键一致性问题：

- 原来前端可以编辑用户角色
- 但后端更新用户时会把 `role` 清空
- 导致页面看似可改角色，实际不生效

现在已经改为：

- 新增用户时校验角色合法性
- 编辑用户时允许角色真实保存
- 同时增加手机号唯一性校验

### 3.7 活动管理与流动人口导出入口补齐

后端原本已经存在以下导出接口：

- `/export/activity`
- `/export/floating`

本轮在前端页面补上了对应导出按钮，使导出能力真正可达。

### 3.8 消息中心详情体验增强

扩展后，消息中心不再只是“已读/未读管理页”，而是具备了完整的消息查看体验。

## 4. 对原有逻辑做出的修改

本轮不仅新增功能，也对原有逻辑做了多处修正，以下是核心变更点。

### 4.1 修改数据库初始化方式

原有方式：

- 依赖多个 Sprint 增量 SQL 逐个执行

修改后：

- 统一切换为执行 `community_service.sql` 完成完整初始化

影响：

- 环境搭建更简单
- 更适合项目交付和回归验收

### 4.2 修复数据库演示数据错位问题

原有问题：

- SQL 中残留导出后的 `AUTO_INCREMENT` 起始值
- 导致用户、工单、评价、积分等演示数据主键错位
- 出现“数据库有数据，但用户看不到自己的数据”的现象

修改后：

- 统一修正总脚本中的 `AUTO_INCREMENT` 起始值
- 修正工单超时演示更新语句的命中方式

影响：

- 演示数据关系恢复正确
- “我的工单、我的评价、积分、消息”链路恢复正常

### 4.3 修复分页总数异常

原有问题：

- 部分接口虽然返回 `records`
- 但 `total = 0`

修改后：

- 新增 MyBatis-Plus 分页拦截器配置：
  - `houduan/src/main/java/com/community/config/MybatisPlusConfig.java`

影响：

- 分页接口的总数、页数恢复正常
- 前端分页组件可正确显示总条数

### 4.4 修复用户缓存导致的旧用户 ID 问题

原有问题：

- `SysUserService` 对登录查用户和按 ID 查用户使用缓存
- 当数据库重建后，Redis 里仍可能保留旧缓存
- 登录后会返回旧用户 ID
- 从而导致“当前登录用户”和数据库真实数据对不上

修改后：

- 去掉登录核心查询对用户缓存的依赖

影响：

- 重建数据库后重新登录即可拿到正确用户 ID
- 真实环境回归更稳定

### 4.5 修改公告发布逻辑

原有逻辑：

- 公告发布时仅保存公告
- 订阅推送逻辑未真正完成

修改后：

- 公告发布时进入 `publishAndNotify`
- 按订阅设置筛选接收用户
- 自动发送站内消息

影响：

- 公告模块从单纯内容发布升级为“发布 + 触达”闭环

### 4.6 修改工单接口返回内容

原有问题：

- 普通用户工单列表中责任人名称依赖前端单独查用户列表
- 在非管理员场景容易回退成 `用户ID`

修改后：

- `WorkOrder` 新增展示字段 `assigneeName`
- 后端在列表和详情接口中统一回填责任人姓名

影响：

- 普通用户也能直接看到责任人姓名
- 页面显示更稳定、语义更清晰

### 4.7 修改用户管理更新逻辑

原有问题：

- 编辑用户时角色不保存
- 手机号重复未严格校验

修改后：

- 编辑用户允许真实更新角色
- 新增与编辑时都校验手机号唯一性
- 增加角色合法性校验

影响：

- 用户管理更符合前端页面预期
- 避免角色修改“假生效”
